ИНФОРМАЦИОННАЯ СИСТЕМА УПРАВЛЕНИЯ IT-ПРОЕКТАМИ «TANKATUS IS»

Разработчик: Лучина А.С.
Платформа: iOS (iPhone / iPad), iOS 17+
Язык программирования: Swift 5.9
Фреймворк пользовательского интерфейса: SwiftUI
Дата: 07.05.2026


1. НАЗНАЧЕНИЕ И ОБЛАСТЬ ПРИМЕНЕНИЯ

1.1. Назначение системы

Информационная система «Tankatus IS» предназначена для автоматизации процессов управления IT-проектами в организации. Система обеспечивает полный цикл управления проектной деятельностью — от инициации проекта до контроля исполнения задач и анализа ключевых показателей эффективности.

1.2. Решаемые задачи

Система решает следующие задачи:
— планирование IT-проектов с указанием сроков, бюджетов и приоритетов;
— декомпозиция работ на задачи и подзадачи с неограниченной вложенностью в соответствии с методологией WBS (Work Breakdown Structure);
— распределение задач между исполнителями с контролем нагрузки;
— отслеживание статусов выполнения задач в реальном времени;
— визуализация хода работ в форматах Kanban-доска, иерархический список и диаграмма Ганта;
— мониторинг ключевых показателей эффективности по методологии Earned Value Management (EVM);
— контроль нагрузки сотрудников и распределения трудозатрат;
— ролевое разграничение доступа к функциям системы.

1.3. Целевая аудитория

Система ориентирована на три категории пользователей. Руководители проектов используют систему для создания проектов, назначения задач, контроля сроков и бюджета. Разработчики и аналитики используют систему для просмотра назначенных задач, изменения статусов и учёта трудозатрат. Администраторы осуществляют полное управление всеми данными системы.


2. ТЕХНИЧЕСКИЙ СТЕК И АРХИТЕКТУРА

2.1. Используемые технологии

Приложение разработано на языке программирования Swift версии 5.9 для платформы iOS 17 и выше. Пользовательский интерфейс построен с использованием декларативного фреймворка SwiftUI. Для визуализации данных (графики, диаграммы) используется библиотека Swift Charts, входящая в состав iOS SDK. Управление зависимостями и генерация проекта Xcode выполняется с помощью инструмента XcodeGen. Данные хранятся в оперативной памяти устройства (in-memory storage), сессия пользователя сохраняется через UserDefaults.

2.2. Архитектурный паттерн

Приложение построено по архитектурному паттерну MVVM (Model — View — ViewModel). Слой Model представлен структурами данных (User, Project, ProjectTask, Stage, KpiSnapshot и другие). Слой View реализован декларативно на SwiftUI и отвечает исключительно за отображение данных. Слой ViewModel содержит бизнес-логику, управляет состоянием экранов и обеспечивает реактивное обновление интерфейса через механизм Combine (@Published, ObservableObject). Хранилище данных реализовано в виде синглтона LocalDataStore, предоставляющего CRUD-операции и вычисляемую аналитику.

2.3. Режим работы

Приложение работает полностью автономно в offline-режиме. Все данные хранятся в оперативной памяти устройства и инициализируются предзаполненными демонстрационными данными при первом запуске. Сетевое подключение и серверная часть не требуются. Изменения, внесённые пользователем, сохраняются в рамках одной сессии работы приложения. Сессия авторизации сохраняется между запусками через UserDefaults.


3. СТРУКТУРА ПРОЕКТА

Исходный код приложения организован следующим образом. Точка входа — файл TankatusApp.swift, содержащий структуру с атрибутом @main. Модели данных расположены в каталоге Models и содержат десять структур данных. Сервисный слой расположен в каталоге Services и включает три файла: AuthStore (управление аутентификацией), LocalDataStore (хранилище данных и бизнес-логика) и WebSocketService (заглушка для offline-режима). ViewModels расположены в одном файле ViewModels.swift и включают семь классов ViewModel, а также рекурсивную структуру TaskNode. Представления (Views) организованы по модулям: Auth (экран входа), Dashboard (главный дашборд), Projects (список проектов, детальный вид, форма, диаграмма Ганта), Tasks (мои задачи, форма задачи), KPI (аналитика EVM), Users (команда и нагрузка), Profile (профиль пользователя), Components (переиспользуемые компоненты). Утилиты расположены в каталоге Utilities и содержат константы и расширения форматтеров дат.

Общий объём кодовой базы составляет приблизительно 2500 строк Swift-кода в 16 файлах.


4. РОЛЕВАЯ МОДЕЛЬ И СИСТЕМА ПРАВ ДОСТУПА

4.1. Определение ролей

В системе реализована ролевая модель доступа с тремя уровнями прав. Роль «Администратор» (идентификатор admin) предоставляет полный доступ ко всем функциям и данным системы. Роль «Менеджер» (идентификатор manager) позволяет создавать, редактировать и удалять проекты и задачи. Роль «Сотрудник» (идентификатор worker) ограничивает пользователя просмотром проектов, в которых он является участником, и изменением статуса только назначенных ему задач.

4.2. Матрица доступа

Создание, редактирование и удаление проектов доступно только администраторам и менеджерам. Просмотр всех проектов доступен администраторам и менеджерам, сотрудники видят только проекты, в которых состоят участниками. Создание, редактирование и удаление задач доступно администраторам и менеджерам. Создание подзадач доступно администраторам и менеджерам. Изменение статуса задачи доступно администраторам, менеджерам и исполнителю задачи (только для назначенной ему задачи). Просмотр дашборда, KPI-аналитики, списка пользователей и профиля доступен всем авторизованным пользователям.

4.3. Реализация проверки прав

Проверка роли реализована через вычисляемое свойство canManage, которое возвращает true для ролей admin и manager. Для задач дополнительно проверяется, является ли текущий пользователь исполнителем задачи (поле assignee_id совпадает с идентификатором текущего пользователя).


5. ФУНКЦИОНАЛЬНЫЙ МОДУЛЬ «АУТЕНТИФИКАЦИЯ»

5.1. Экран входа

Экран входа представляет собой форму с градиентным синим фоном, логотипом приложения и двумя полями ввода. Поле «Email» имеет тип клавиатуры emailAddress с отключённой автокапитализацией. Поле «Пароль» использует защищённый ввод SecureField. Под полями расположена кнопка «Войти», которая во время обработки запроса отображает индикатор загрузки и текст «Вход...». В случае ошибки под полями появляется красный блок с описанием ошибки.

5.2. Логика аутентификации

При нажатии кнопки «Войти» система выполняет следующую последовательность действий. Проверяет, что оба поля не пустые. Ищет пользователя по введённому email в локальном хранилище данных. Если пользователь не найден, отображает ошибку «Пользователь не найден». Проверяет введённый пароль. Если пароль не совпадает, отображает ошибку «Неверный пароль». При успешной проверке устанавливает текущего пользователя и сохраняет токен сессии в UserDefaults.

5.3. Сохранение сессии

Токен сессии сохраняется в UserDefaults в формате local_ плюс идентификатор пользователя. При повторном запуске приложения система автоматически проверяет наличие сохранённого токена и восстанавливает сессию без необходимости повторного ввода учётных данных. Если пользователь с указанным идентификатором не найден, токен очищается и отображается экран входа.

5.4. Выход из системы

Функция выхода очищает токен авторизации, сбрасывает текущего пользователя и удаляет соответствующий ключ из UserDefaults. Пользовательский интерфейс автоматически переключается на экран входа.


6. ФУНКЦИОНАЛЬНЫЙ МОДУЛЬ «ДАШБОРД»

6.1. Описание

Дашборд является главным экраном системы после входа и обеспечивает быстрый обзор текущего состояния проектной деятельности. Расположен на первой вкладке нижней панели навигации с иконкой графика.

6.2. Сводная статистика

В верхней части экрана расположена сетка из четырёх карточек статистики. Первая карточка «Всего проектов» отображает общее количество проектов, доступных текущему пользователю (синий цвет). Вторая карточка «Активных» показывает количество проектов со статусом «Активный» (зелёный цвет). Третья карточка «Всего задач» отображает суммарное количество задач в системе (фиолетовый цвет). Четвёртая карточка «Просрочено задач» показывает количество задач, у которых прошёл срок выполнения и статус не равен «Готово» (красный цвет).

6.3. Секция активных проектов

Ниже статистики расположен список из до пяти активных и планируемых проектов. Каждый проект отображается карточкой, содержащей название, бейдж приоритета, прогрессбар с процентом выполнения, бейдж статуса и дату окончания. При нажатии на карточку происходит переход к детальному виду проекта.

6.4. Диаграмма задач по статусам

В нижней части экрана расположена кольцевая секторная диаграмма, визуализирующая распределение всех задач по статусам. Используется тип SectorMark библиотеки Swift Charts с внутренним радиусом 50 процентов. Легенда отображается автоматически.

6.5. Обновление данных

Экран поддерживает обновление данных жестом Pull-to-Refresh, а также автоматически загружает данные при первом отображении.


7. ФУНКЦИОНАЛЬНЫЙ МОДУЛЬ «ПРОЕКТЫ»

7.1. Список проектов

Экран списка проектов расположен на второй вкладке с иконкой папки. Отображает все проекты, доступные текущему пользователю: администраторы и менеджеры видят все проекты, сотрудники — только те, в которых являются участниками.

7.2. Поиск

В верхней части экрана встроено поле поиска, осуществляющее фильтрацию проектов по названию в реальном времени при вводе текста. Поиск регистронезависимый.

7.3. Фильтрация по статусу

В правой части панели инструментов расположено меню фильтрации по статусу проекта. Доступные варианты: Все (показывает все проекты), Планирование, Активные, На паузе, Завершённые. Иконка фильтра визуально меняется при активном фильтре.

7.4. Карточка проекта

Каждый проект отображается карточкой со следующими элементами. Заголовок: название проекта и бейдж приоритета. Описание: до двух строк текста серым цветом. Прогрессбар: индикатор выполнения, окрашенный в зелёный при 100 процентах. Счётчик задач: формат «X/Y задач» слева, бейдж статуса справа. Подвал: иконка с именем менеджера и дата окончания, выделенная красным при просрочке. Карточки имеют белый фон, закруглённые углы 14 пунктов и лёгкую тень.

7.5. Создание проекта

Для пользователей с ролями «Администратор» и «Менеджер» в левой части панели инструментов отображается кнопка со знаком плюс. При нажатии открывается форма создания нового проекта в виде модального окна (sheet).

7.6. Редактирование и удаление проекта

Для пользователей с ролями «Администратор» и «Менеджер» доступны два способа доступа к действиям. Первый — долгое нажатие на карточку проекта вызывает контекстное меню с пунктами «Редактировать» и «Удалить». Второй — свайп влево по карточке открывает красную кнопку «Удалить» и оранжевую кнопку «Изменить». При удалении проекта также удаляются все связанные задачи, этапы и KPI-снимки.

7.7. Форма создания и редактирования проекта

Форма проекта состоит из трёх секций. Секция «Основное» содержит обязательное текстовое поле «Название проекта» и необязательное многострочное поле «Описание». Секция «Параметры» содержит выбор статуса (Планирование, Активный, На паузе, Завершён), выбор приоритета (Низкий, Средний, Высокий, Критический) и числовое поле «Бюджет» в рублях. Секция «Сроки» содержит выбор даты начала и даты окончания. В панели инструментов формы расположены кнопки «Отмена» (закрытие без сохранения) и «Сохранить» (валидация и сохранение). Кнопка «Сохранить» неактивна, если название проекта пустое.

7.8. Статусы проектов

В системе определены четыре статуса проекта. Статус «Планирование» (код planning) обозначает проект на стадии подготовки, отображается серым бейджем. Статус «Активный» (код active) обозначает проект с ведущейся работой, отображается зелёным бейджем. Статус «На паузе» (код on_hold) обозначает временную приостановку работ, отображается оранжевым бейджем. Статус «Завершён» (код completed) обозначает успешное завершение проекта, отображается синим бейджем.

7.9. Приоритеты проектов

Определены четыре уровня приоритета. «Низкий» (код low, серый бейдж), «Средний» (код medium, синий бейдж), «Высокий» (код high, оранжевый бейдж), «Критический» (код critical, красный бейдж).


8. ФУНКЦИОНАЛЬНЫЙ МОДУЛЬ «ДЕТАЛЬНЫЙ ВИД ПРОЕКТА»

8.1. Описание

Детальный вид проекта открывается при нажатии на карточку проекта в списке. Экран содержит шапку с общей информацией и четыре вкладки, переключаемые сегментированным элементом управления: Kanban, Список, Гант, Участники.

8.2. Шапка проекта

Шапка всегда отображается над вкладками и содержит следующие элементы. Первая строка: бейджи статуса и приоритета проекта, дата окончания с иконкой календаря (красная при просрочке). Вторая строка: сетка из четырёх мини-показателей — количество задач верхнего уровня, количество выполненных задач, количество просроченных задач, процент прогресса. Третья строка: горизонтальный прогрессбар проекта, окрашенный в зелёный при 100 процентах.

8.3. Вкладка «Kanban»

Вкладка Kanban представляет собой горизонтально прокручиваемую доску с четырьмя колонками. Колонка «К выполнению» содержит задачи со статусом todo. Колонка «В работе» содержит задачи со статусом in_progress. Колонка «На ревью» содержит задачи со статусом review. Колонка «Готово» содержит задачи со статусом done. Каждая колонка имеет ширину 210 пунктов, заголовок с названием и счётчик задач.

Карточка задачи в Kanban содержит: название задачи (до трёх строк), меню действий (три точки), прогресс подзадач при наличии (в формате «X/Y» с мини-прогрессбаром), имя исполнителя, бейдж приоритета, срок выполнения и кнопку быстрой смены статуса. Меню действий для администраторов и менеджеров включает пункты «Редактировать», «Добавить подзадачу» и «Удалить». Для сотрудников-исполнителей доступен только пункт «Редактировать».

8.4. Вкладка «Список»

Вкладка «Список» отображает задачи в виде рекурсивного иерархического дерева с неограниченной глубиной вложенности. Каждый уровень вложенности визуально обозначен отступом в 20 пунктов и вертикальной синей линией слева шириной 2 пикселя. Задачи, имеющие дочерние элементы, отображают кнопку раскрытия/свёртывания в виде треугольника (стрелка вправо — свёрнуто, стрелка вниз — развёрнуто). Листовые задачи отображают маленькую точку вместо кнопки раскрытия.

Каждый элемент списка содержит: кнопку раскрытия, название задачи (полужирное для верхнего уровня, обычное для подзадач), описание (одна строка), бейджи статуса и приоритета, срок выполнения, имя исполнителя, прогресс потомков при наличии дочерних задач. Меню действий (три точки) доступно для каждой задачи и содержит пункты в зависимости от роли пользователя.

Фон элементов различается по уровню вложенности: корневые задачи имеют белый фон с выраженной тенью, подзадачи — слегка затемнённый фон с минимальной тенью.

8.5. Вкладка «Гант»

Вкладка «Гант» отображает полнофункциональную интерактивную диаграмму Ганта. Подробное описание приведено в разделе 9.

8.6. Вкладка «Участники»

Вкладка «Участники» отображает список членов команды проекта. Для каждого участника показывается аватар с инициалами, полное имя, адрес электронной почты, название отдела и роль в проекте (менеджер или участник) в виде цветного бейджа.

8.7. Создание задачи

Для администраторов и менеджеров в правом верхнем углу панели инструментов отображается кнопка со знаком плюс. При нажатии открывается форма создания новой задачи верхнего уровня.


9. ФУНКЦИОНАЛЬНЫЙ МОДУЛЬ «ДИАГРАММА ГАНТА»

9.1. Описание

Диаграмма Ганта представляет собой интерактивную временную шкалу, отображающую задачи проекта в виде горизонтальных полос. Диаграмма поддерживает масштабирование, иерархическое отображение задач и подзадач, а также визуальное выделение текущего дня, выходных и просроченных задач.

9.2. Структура экрана

Экран состоит из трёх частей. Верхняя панель содержит заголовок «Диаграмма Ганта» и кнопки масштабирования (лупа с минусом и лупа с плюсом). Основная область разделена на левую панель шириной 200 пунктов (список задач) и правую область (временная шкала с барами). Нижняя панель содержит легенду с цветовыми обозначениями статусов.

9.3. Масштабирование

Диаграмма поддерживает масштабирование от 14 до 56 пикселей на один день с шагом 7 пикселей. По умолчанию масштаб составляет 28 пикселей на день. Масштабирование выполняется кнопками в верхней панели.

9.4. Левая панель

Левая панель отображает иерархический список задач. Подзадачи обозначены символом └ с отступом 12 пунктов на каждый уровень вложенности. Перед названием каждой задачи отображается цветная точка, соответствующая приоритету (серая — низкий, синяя — средний, оранжевая — высокий, красная — критический). Справа от названия отображается текстовый статус задачи.

9.5. Временная шкала

Заголовок временной шкалы содержит два уровня: названия месяцев (на русском языке) и номера дней. Фон шкалы визуально выделяет выходные дни (светло-серый), текущий день (голубой фон + пунктирная синяя вертикальная линия). Строки чередуются для удобства чтения.

9.6. Бары задач

Каждая задача отображается горизонтальным баром. Позиция бара по горизонтали определяется датой начала задачи. Ширина бара определяется периодом от даты начала до даты окончания. Минимальная ширина бара равна ширине одного дня. Цвет рамки бара соответствует статусу задачи: серый для «К выполнению», синий для «В работе», фиолетовый для «На ревью», зелёный для «Готово». Внутри бара отображается заливка прогресса: 0 процентов для todo, 50 процентов для in_progress, 80 процентов для review, 100 процентов для done. Просроченные задачи выделяются красной рамкой и красноватым фоном. При достаточной ширине бара (более 50 пунктов) внутри отображается название задачи.

9.7. Легенда

Под диаграммой расположена горизонтально прокручиваемая легенда с цветными прямоугольниками для каждого статуса и синей линией для обозначения текущего дня.


10. ФУНКЦИОНАЛЬНЫЙ МОДУЛЬ «УПРАВЛЕНИЕ ЗАДАЧАМИ»

10.1. Иерархическая структура задач

Система поддерживает структуру разбивки работ (Work Breakdown Structure) с неограниченной глубиной вложенности подзадач. Каждая задача может иметь произвольное количество дочерних задач. Каждая дочерняя задача может, в свою очередь, иметь свои дочерние задачи. Связь между родительской и дочерней задачей реализована через поле parent_task_id. Задачи верхнего уровня имеют parent_task_id равное nil.

Подзадача является полноценной задачей со всеми теми же атрибутами: название, описание, статус, приоритет, исполнитель, этап, оценка трудозатрат, срок выполнения. Ограничений на глубину вложенности нет.

10.2. Агрегация прогресса

Прогресс родительской задачи вычисляется рекурсивно на основе всех потомков на всех уровнях вложенности. Формула: количество выполненных потомков делится на общее количество потомков. Например, если задача имеет две подзадачи, одна из которых имеет три собственные подзадачи, то общее количество потомков равно пяти, и прогресс вычисляется как отношение выполненных из пяти.

10.3. Рекурсивная структура данных

Для представления дерева задач используется рекурсивный тип TaskNode, реализованный как indirect enum в Swift. Это позволяет описать дерево произвольной глубины без ограничений. Каждый узел содержит задачу и массив дочерних узлов. Вычисляемые свойства узла включают: общее количество потомков (totalCount), количество выполненных потомков (doneCount) и процент прогресса (subtaskProgress).

10.4. Создание подзадачи

Создание подзадачи доступно через меню действий (три точки) любой задачи — пункт «Добавить подзадачу». Функция доступна для администраторов и менеджеров. При создании подзадачи открывается та же форма задачи, но с дополнительной секцией в шапке, отображающей название родительской задачи. Этап подзадачи по умолчанию наследуется от родительской задачи. Создание подзадачи доступно для задач любого уровня вложенности, включая подзадачи.

10.5. Форма задачи

Для создания задачи, создания подзадачи и редактирования задачи используется единая форма. Заголовок формы автоматически меняется: «Новая задача» для задач верхнего уровня, «Новая подзадача» для дочерних задач, «Редактировать задачу» для редактирования.

Форма содержит следующие секции и поля. Секция родительской задачи (отображается только при создании подзадачи): иконка и текст «Подзадача для:» с названием родителя. Секция «Основное»: обязательное текстовое поле «Название», необязательное многострочное поле «Описание». Секция «Параметры»: выбор статуса (К выполнению, В работе, На ревью, Готово), выбор приоритета (Низкий, Средний, Высокий, Критический), числовое поле «Оценка в часах». Секция «Исполнитель и этап»: выбор исполнителя из списка участников проекта (с вариантом «Не назначен»), выбор этапа из списка этапов проекта (с вариантом «Без этапа»). Секция «Срок»: выбор даты окончания с возможностью сброса.

10.6. Статусы задач

В системе определены четыре статуса задачи. Статус «К выполнению» (код todo) обозначает задачу, ожидающую начала работы, отображается серым бейджем. Статус «В работе» (код in_progress) обозначает задачу в активной разработке, отображается синим бейджем. Статус «На ревью» (код review) обозначает задачу на проверке, отображается фиолетовым бейджем. Статус «Готово» (код done) обозначает выполненную задачу, отображается зелёным бейджем.

10.7. Приоритеты задач

Определены четыре уровня приоритета задач. «Низкий» (код low, серый бейдж), «Средний» (код medium, синий бейдж), «Высокий» (код high, оранжевый бейдж), «Критический» (код critical, красный бейдж).

10.8. Быстрая смена статуса

Смена статуса задачи доступна несколькими способами. В Kanban: меню «Статус» с вариантами «В работе», «На ревью», «Готово». В иерархическом списке: подменю «Изменить статус» с вариантами «К выполнению», «В работе», «На ревью», «Готово». В разделе «Мои задачи»: последовательные кнопки действий — «В работу» для задач с статусом todo, «На ревью» для задач в работе, «Готово» для задач на ревью.

10.9. Удаление задачи

При удалении задачи также удаляются все её подзадачи на всех уровнях. Счётчики задач проекта автоматически пересчитываются. Функция удаления доступна только администраторам и менеджерам.


11. ФУНКЦИОНАЛЬНЫЙ МОДУЛЬ «МОИ ЗАДАЧИ»

11.1. Описание

Раздел «Мои задачи» предоставляет персональный список всех задач, назначенных текущему пользователю, вне зависимости от принадлежности к проекту. Расположен на третьей вкладке нижней панели навигации с иконкой галочки.

11.2. Фильтрация

Экран предоставляет два способа фильтрации. Первый — горизонтальная панель чипов под заголовком: «Все», «К выполнению», «В работе», «На ревью». Активный чип окрашен в синий цвет с белым текстом, неактивные — белый фон с тёмным текстом. Второй — меню фильтрации в панели инструментов с теми же вариантами.

11.3. Карточка задачи

Каждая задача отображается карточкой, содержащей следующие элементы. Проект и этап: иконка папки и название проекта синим цветом, через косую черту — название этапа серым цветом. Название задачи: полужирным шрифтом. Описание: до двух строк серым цветом (при наличии). Информационная строка: бейдж статуса, бейдж приоритета, трудозатраты в формате «X/Y ч» (фактические/плановые), дата окончания (красная при просрочке). Кнопки действий: отображаются для задач, которые ещё не выполнены, и позволяют переместить задачу на следующий статус.

Кнопки действий реализованы в виде капсул с полупрозрачным фоном цвета. Для задач со статусом «К выполнению» — кнопка «В работу» (синяя). Для задач «В работе» — кнопка «На ревью» (фиолетовая). Для задач «На ревью» — кнопка «Готово» (зелёная).

11.4. Обновление данных

Экран поддерживает обновление данных жестом Pull-to-Refresh и автоматически загружает задачи при первом отображении.


12. ФУНКЦИОНАЛЬНЫЙ МОДУЛЬ «KPI / МЕТОД ОСВОЕННОГО ОБЪЁМА»

12.1. Описание

Модуль реализует мониторинг эффективности проектов по методологии Earned Value Management (EVM) — международному стандарту измерения производительности проектов. Расположен на четвёртой вкладке нижней панели навигации с иконкой графика.

12.2. Выбор проекта

В верхней части экрана расположен выпадающий список для выбора проекта. Отображаются только проекты со статусами «Активный» и «Планирование». При смене проекта автоматически загружаются соответствующие KPI-снимки. По умолчанию выбирается первый доступный проект.

12.3. KPI-индикаторы

Под выбором проекта расположена сетка из четырёх карточек показателей.

Показатель SPI (Schedule Performance Index — индекс выполнения расписания) вычисляется как отношение освоенного объёма к плановому объёму (EV / PV). При значении больше или равном единице отображается зелёным цветом с иконкой галочки и подписью «В норме». При значении менее единицы отображается красным цветом с иконкой предупреждения и подписью «Отклонение».

Показатель CPI (Cost Performance Index — индекс выполнения стоимости) вычисляется как отношение освоенного объёма к фактической стоимости (EV / AC). Правила отображения аналогичны SPI.

Показатель CV (Cost Variance — отклонение по стоимости) вычисляется как разность освоенного объёма и фактической стоимости (EV − AC). Положительное значение отображается зелёным, отрицательное — красным. Единица измерения: BYN (белорусский рубль).

Показатель SV (Schedule Variance — отклонение по расписанию) вычисляется как разность освоенного объёма и планового объёма (EV − PV). Правила отображения аналогичны CV.

Каждая карточка визуально выделена рамкой (зелёной при нормальном значении, красной при отклонении) и содержит заголовок показателя, числовое значение, подзаголовок с расшифровкой и текст состояния.

12.4. EVM-график

Ниже индикаторов расположен линейный график, визуализирующий динамику трёх ключевых метрик EVM по временным снимкам. Линия PV (Planned Value — плановый объём) отображается серым пунктиром. Линия EV (Earned Value — освоенный объём) отображается сплошной синей линией. Линия AC (Actual Cost — фактическая стоимость) отображается сплошной красной линией. На каждой точке данных отображается кружок-маркер. Ось X представляет даты снимков, ось Y — значение в рублях. Высота графика составляет 220 пунктов. Легенда отображается автоматически.

12.5. Прогнозные показатели

В нижней части экрана расположены две карточки прогнозных показателей. Показатель EAC (Estimate at Completion — прогнозная стоимость завершения) вычисляется как BAC / CPI и показывает ожидаемую итоговую стоимость проекта при текущей эффективности. Отображается оранжевым цветом. Показатель ETC (Estimate to Complete — стоимость оставшихся работ) вычисляется как EAC − AC и показывает, сколько ещё потребуется вложить для завершения проекта. Отображается фиолетовым цветом.

12.6. Интерпретация показателей

Значения SPI и CPI более единицы означают, что проект опережает план (экономия ресурсов). Значение, равное единице, означает точное соответствие плану. Значение менее единицы означает отставание от плана (перерасход ресурсов). Положительные значения CV и SV означают положительное отклонение (экономия или опережение). Нулевые — точное соответствие плану. Отрицательные — негативное отклонение (перерасход или отставание).


13. ФУНКЦИОНАЛЬНЫЙ МОДУЛЬ «КОМАНДА»

13.1. Описание

Раздел «Команда» отображает всех пользователей системы с информацией о текущей нагрузке. Доступен всем авторизованным пользователям.

13.2. Список пользователей

Экран отображает список всех зарегистрированных пользователей. В верхней части расположено поле поиска, осуществляющее фильтрацию по имени и адресу электронной почты. Экран поддерживает обновление данных жестом Pull-to-Refresh.

13.3. Карточка пользователя

Каждый пользователь отображается карточкой, состоящей из двух частей. Верхняя часть содержит: аватар с инициалами (цвет зависит от роли: красный для администратора, синий для менеджера, зелёный для разработчика, оранжевый для аналитика), полное имя, бейдж «Неактивен» при неактивном аккаунте, адрес электронной почты, роль в виде цветного бейджа и название отдела.

Нижняя часть содержит показатели нагрузки. Три метрики отображаются в одну строку: «Плановые ч» — сумма оценок трудозатрат всех назначенных задач, «Фактические ч» — сумма фактических трудозатрат выполненных задач, «Задач» — количество активных задач (со статусом, отличным от «Готово»). Ниже отображается прогрессбар «Загрузка», вычисляемый как отношение фактических часов к плановым. При загрузке более 110 процентов прогрессбар и число окрашиваются в красный цвет, обозначая перегрузку.


14. ФУНКЦИОНАЛЬНЫЙ МОДУЛЬ «ПРОФИЛЬ»

14.1. Описание

Экран профиля отображает информацию о текущем авторизованном пользователе и предоставляет функцию выхода из системы. Расположен на пятой вкладке нижней панели навигации.

14.2. Состав экрана

Экран содержит следующие элементы. Аватар: большой круглый блок размером 80 пунктов с инициалами пользователя на градиентном фоне. Имя и email: полное имя крупным полужирным шрифтом, адрес электронной почты серым цветом ниже.

Секция «Информация» оформлена в виде формы и содержит: роль пользователя (цветной бейдж: «Администратор», «Менеджер» или «Работник»), название отдела, статус активности (бейдж «Активен» зелёного цвета).

Секция «Система» отображает текущий режим работы приложения. Показывается текст «Режим» с индикатором: зелёный кружок и подпись «Локальный режим», обозначающая, что приложение работает автономно без сервера.

Секция «Выход» содержит единственную кнопку «Выйти из аккаунта», оформленную красным цветом как деструктивное действие. При нажатии вызывается функция выхода из системы.


15. МОДЕЛИ ДАННЫХ

15.1. Модель User (Пользователь)

Структура содержит следующие поля: идентификатор (id, строка), адрес электронной почты (email, строка), полное имя (full_name, строка), роль (role, строка — admin, manager или worker), название отдела (department, строка, необязательное), URL аватара (avatar_url, строка, необязательное), признак активности (is_active, логическое), дата создания (created_at, строка). Вычисляемые свойства: отображаемая роль на русском языке (displayRole), инициалы из первых букв имени и фамилии (initials).

15.2. Модель Project (Проект)

Структура содержит: идентификатор, название, описание (необязательное), статус (planning, active, on_hold, completed), приоритет (low, medium, high, critical), дата начала (необязательная), дата окончания (необязательная), бюджет в рублях (необязательный), идентификатор менеджера, имя менеджера, количество всех задач, количество выполненных задач, массив участников, дата создания, дата обновления. Вычисляемые свойства: прогресс выполнения (progress — отношение done_tasks к total_tasks), признак просрочки (isOverdue — дата окончания прошла и статус не равен completed).

15.3. Модель ProjectMember (Участник проекта)

Структура содержит: идентификатор, полное имя, адрес электронной почты, роль в системе, отдел (необязательное), роль в проекте (project_role — manager или member).

15.4. Модель Stage (Этап)

Структура содержит: идентификатор, идентификатор проекта (project_id), название, описание (необязательное), порядковый индекс (order_index), статус (planned, in_progress, completed), дата начала (необязательная), дата окончания (необязательная).

15.5. Модель ProjectTask (Задача)

Структура содержит: идентификатор, идентификатор проекта, идентификатор этапа (необязательный), название этапа (необязательное), идентификатор родительской задачи (parent_task_id, необязательный — nil для задач верхнего уровня), название, описание (необязательное), статус (todo, in_progress, review, done), приоритет (low, medium, high, critical), идентификатор исполнителя (необязательный), имя исполнителя (необязательное), имя создателя (необязательное), плановые трудозатраты в часах (необязательные), фактические трудозатраты в часах (необязательные), дата начала (необязательная), срок выполнения (необязательный), название проекта (необязательное), массив комментариев, дата создания, дата обновления. Вычисляемое свойство: признак просрочки (isOverdue — срок прошёл и статус не равен done).

Поле parent_task_id реализует самоссылку (self-reference), позволяя строить иерархию задач произвольной глубины.

15.6. Модель KpiSnapshot (Снимок KPI)

Структура содержит: идентификатор, идентификатор проекта, дата снимка, плановый объём (planned_value), освоенный объём (earned_value), фактическая стоимость (actual_cost), бюджет по завершении (budget_at_completion, необязательный), индекс SPI (необязательный), индекс CPI (необязательный), отклонение CV (необязательное), отклонение SV (необязательное), прогноз EAC (необязательный), прогноз ETC (необязательный).

15.7. Модель DashboardStats (Статистика дашборда)

Структура содержит: массив статистики по проектам (project_stats), массив статистики по задачам (task_stats), количество просроченных задач (overdue_tasks), массив последних активных проектов (recent_active_projects).

15.8. Модель UserLoad (Нагрузка пользователя)

Структура содержит: идентификатор, полное имя, отдел (необязательное), плановые часы (total_planned, необязательное), фактические часы (total_actual, необязательное), количество активных задач (active_tasks, необязательное). Вычисляемое свойство: инициалы (initials).

15.9. Связи между моделями

Один пользователь может быть участником нескольких проектов через модель ProjectMember. Один проект содержит множество этапов (Stage). Один проект содержит множество задач (ProjectTask). Одна задача может иметь множество дочерних задач через self-reference по полю parent_task_id. Один проект имеет множество KPI-снимков (KpiSnapshot).


16. ПЕРЕИСПОЛЬЗУЕМЫЕ UI-КОМПОНЕНТЫ

16.1. StatusBadge

Компонент для отображения статуса в виде цветной капсулы. Принимает код статуса и автоматически определяет текст и цвет. Поддерживает статусы проектов (planning, active, on_hold, completed) и задач (todo, in_progress, review, done).

16.2. PriorityBadge

Компонент для отображения приоритета в виде цветной капсулы. Принимает код приоритета и автоматически определяет текст и цвет: серый для low, синий для medium, оранжевый для high, красный для critical.

16.3. StatCard

Карточка статистики с заголовком, крупным числовым значением, необязательной подписью и цветовым акцентом. Имеет белый фон, закруглённые углы 12 пунктов и тень.

16.4. InitialsAvatar

Круглый аватар с инициалами на градиентном цветном фоне. Настраиваемые параметры: текст инициалов, размер (по умолчанию 40 пунктов) и цвет.

16.5. ProgressRow

Горизонтальный прогрессбар с текстовой подписью слева и процентом справа. Окрашивается в зелёный при 100 процентах.

16.6. KPICard

Специализированная карточка для отображения KPI-показателя. Содержит заголовок, подзаголовок с описанием, крупное значение, иконку состояния (галочка или предупреждение) и текст «В норме» или «Отклонение». Рамка карточки окрашена в зелёный или красный цвет в зависимости от значения.

18. НАВИГАЦИЯ ПРИЛОЖЕНИЯ

18.1. Нижняя панель навигации

Приложение использует нижнюю панель навигации (TabView) с пятью вкладками. Первая вкладка «Дашборд» с иконкой графика — главный экран со сводной информацией. Вторая вкладка «Проекты» с иконкой папки — список и управление проектами. Третья вкладка «Задачи» с иконкой галочки — персональные задачи пользователя. Четвёртая вкладка «KPI» с иконкой линейного графика — аналитика по методологии EVM. Пятая вкладка «Профиль» с иконкой пользователя — личный профиль и выход.

18.2. Навигация внутри вкладок

Каждая вкладка использует собственный NavigationStack, обеспечивающий стандартную навигацию iOS со стрелкой «Назад» и вложенными экранами. Из списка проектов осуществляется переход к детальному виду проекта. Из дашборда осуществляется переход к детальному виду проекта через карточки активных проектов.

18.3. Модальные окна

Формы создания и редактирования (проектов, задач, подзадач) открываются в виде модальных окон (sheet), закрываемых кнопкой «Отмена» или жестом свайпа вниз. Каждая форма содержит панель инструментов с кнопками «Отмена» и «Сохранить».
