## 1. Назначение и область применения

### 1.1 Назначение

Информационная система «Tankatus IS» предназначена для автоматизации процессов управления IT-проектами в организации. Система обеспечивает полный цикл управления проектной деятельностью — от инициации проекта до контроля исполнения задач и анализа ключевых показателей эффективности.

### 1.2 Решаемые задачи

- Планирование IT-проектов с указанием сроков, бюджетов, приоритетов
- Декомпозиция работ на задачи и подзадачи с неограниченной вложенностью (WBS — Work Breakdown Structure)
- Распределение задач между исполнителями
- Отслеживание статусов выполнения в реальном времени
- Визуализация хода работ (Kanban-доска, иерархический список, диаграмма Ганта)
- Мониторинг KPI по методологии Earned Value Management (EVM)
- Контроль нагрузки сотрудников
- Ролевое разграничение доступа

### 1.3 Целевая аудитория

| Роль | Использование системы |
|---|---|
| Руководитель проекта | Создание проектов, назначение задач, контроль сроков и бюджета |
| Разработчик / Аналитик | Просмотр своих задач, изменение статусов, учёт трудозатрат |
| Администратор | Полное управление всеми данными системы |

## 2. Технический стек и архитектура

### 2.1 Используемые технологии

| Компонент | Технология | Версия |
|---|---|---|
| Платформа | iOS | 17.0+ |
| Язык программирования | Swift | 5.9 |
| UI-фреймворк | SwiftUI | — |
| Визуализация данных | Swift Charts | iOS 17+ |
| Архитектурный паттерн | MVVM | — |
| Реактивность | Combine | — |
| Управление зависимостями | XcodeGen | 2.x |
| Хранение данных | In-memory (оперативная память) | — |
| Персистентность сессии | UserDefaults | — |

### 2.2 Архитектурный паттерн MVVM

```
┌─────────────┐       ┌──────────────────┐       ┌────────────────┐
│    View     │──────▶│   ViewModel      │──────▶│  LocalDataStore│
│  (SwiftUI)  │◀──────│ (@Published)     │◀──────│  (Singleton)   │
└─────────────┘       └──────────────────┘       └────────────────┘
      │                       │                         │
      │ UI-события            │ Бизнес-логика          │ Данные
      │ @State                │ async/await            │ CRUD-операции
      ▼                       ▼                         ▼
  Пользователь          ObservableObject          Seed + Мутации
```

**View** — декларативное описание UI на SwiftUI. Подписывается на изменения ViewModel через `@StateObject` / `@ObservedObject`.

**ViewModel** — содержит `@Published`-свойства, вызывает методы LocalDataStore, управляет состоянием загрузки и ошибок.

**LocalDataStore** — синглтон, хранит все данные в памяти, предоставляет CRUD-операции и вычисляемую статистику.

### 2.3 Режим работы

Приложение работает полностью автономно (offline). Все данные хранятся в оперативной памяти устройства и инициализируются предзаполненными данными (seed data) при первом запуске. Изменения сохраняются в рамках одной сессии запуска приложения.

## 4. Ролевая модель и система прав доступа

### 4.1 Определение ролей

| Роль | Идентификатор | Описание |
|---|---|---|
| Администратор | `admin` | Полный доступ ко всем функциям и данным системы |
| Менеджер | `manager` | Создание, редактирование и удаление проектов и задач |
| Сотрудник | `worker` | Просмотр проектов, изменение статуса назначенных задач |

### 4.2 Полная матрица доступа

| Функция | admin | manager | worker |
|---|:---:|:---:|:---:|
| **Проекты** |||
| Просмотр всех проектов | + | + | — |
| Просмотр своих проектов (участник) | + | + | + |
| Создание проекта | + | + | — |
| Редактирование проекта | + | + | — |
| Удаление проекта | + | + | — |
| **Задачи** |||
| Просмотр задач проекта | + | + | + |
| Создание задачи | + | + | — |
| Создание подзадачи | + | + | — |
| Редактирование любой задачи | + | + | — |
| Удаление задачи | + | + | — |
| Изменение статуса назначенной задачи | + | + | + |
| Изменение статуса чужой задачи | + | + | — |
| **Аналитика** |||
| Просмотр дашборда | + | + | + |
| Просмотр KPI / EVM | + | + | + |
| Просмотр списка пользователей | + | + | + |
| Просмотр нагрузки | + | + | + |
| **Профиль** |||
| Просмотр профиля | + | + | + |
| Выход из системы | + | + | + |

### 4.3 Реализация проверки прав

Проверка роли реализована через вычисляемое свойство `canManage` в каждом View:

```swift
private var canManage: Bool {
    let role = authStore.currentUser?.role ?? ""
    return role == "admin" || role == "manager"
}
```

Для задач дополнительно проверяется, является ли текущий пользователь исполнителем:

```swift
private var canEdit: Bool {
    guard let user = authStore.currentUser else { return false }
    if user.role == "admin" || user.role == "manager" { return true }
    return task.assignee_id == user.id
}
```

## 5. Функциональный модуль «Аутентификация»

### 5.1 Описание

Модуль обеспечивает вход пользователя в систему, сохранение сессии и выход.

### 5.2 Экран входа (LoginView)

**Визуальное оформление:**
- Градиентный фон (синий → голубой)
- Логотип приложения (иконка + текст «Танкатус»)
- Подзаголовок «Управление IT-проектами»
- Форма ввода на карточке с закруглёнными углами

**Поля ввода:**
| Поле | Тип | Валидация |
|---|---|---|
| Email | `TextField`, клавиатура `.emailAddress` | Не пустое |
| Пароль | `SecureField` | Не пустое |

**Кнопка «Войти»:**
- При нажатии запускает асинхронный процесс аутентификации
- Во время загрузки отображает `ProgressView` и текст «Вход...»
- Блокируется на время загрузки (`disabled`)

**Обработка ошибок:**
- Красный блок с текстом ошибки под полями ввода
- Текст ошибки: «Пользователь не найден» или «Неверный пароль»

### 5.3 Логика аутентификации (AuthStore)

```
1. Пользователь вводит email и пароль
2. Система ищет пользователя по email в LocalDataStore.users
3. Если не найден → ошибка «Пользователь не найден»
4. Проверяет пароль == "Admin123!"
5. Если не совпадает → ошибка «Неверный пароль»
6. Успех → устанавливает currentUser, сохраняет токен в UserDefaults
7. UI автоматически переключается на MainTabView
```

### 5.4 Персистентность сессии

- Токен формата `local_<userId>` сохраняется в `UserDefaults` по ключу `auth_token`
- При запуске приложения `AuthStore.init()` проверяет наличие сохранённого токена
- Если токен валиден — восстанавливает `currentUser` без повторного входа
- Если пользователь не найден по токену — очищает токен

### 5.5 Выход из системы

- Очистка `token` (nil)
- Очистка `currentUser` (nil)
- Удаление ключа из `UserDefaults`
- UI переключается на `LoginView`

## 6. Функциональный модуль «Дашборд»

### 6.1 Описание

Главный экран системы после входа. Обеспечивает быстрый обзор текущего состояния проектной деятельности.

### 6.2 Навигация

- Первая вкладка в TabView
- Иконка: `chart.bar.fill`
- Заголовок: «Дашборд»
- В правом углу toolbar: аватар текущего пользователя (инициалы)

### 6.3 Сводная статистика

Сетка 2×2 из карточек `StatCard`:

| Позиция | Название | Значение | Цвет |
|---|---|---|---|
| Верхний левый | Всего проектов | Количество проектов пользователя | Синий |
| Верхний правый | Активных | Проекты со статусом `active` | Зелёный |
| Нижний левый | Всего задач | Суммарное количество задач | Фиолетовый |
| Нижний правый | Просрочено задач | Задачи с `isOverdue == true` | Красный |

### 6.4 Секция «Активные проекты»

- Заголовок: «Активные проекты» (`.headline`)
- До 5 проектов со статусами `active` и `planning`
- Каждый проект отображается карточкой `ProjectRowCard`:
  - Название + бейдж приоритета
  - Прогрессбар с процентом выполнения
  - Бейдж статуса + дата окончания
- При нажатии — переход к `ProjectDetailView`

### 6.5 Секция «Задачи по статусам»

- Круговая диаграмма (Donut chart) — `Chart` + `SectorMark`
- Внутренний радиус 50% (кольцевая диаграмма)
- Угловой отступ между секторами: 2pt
- Легенда: автоматическая (`chartLegend(.visible)`)
- Данные: распределение задач по статусам (`task_stats`)

### 6.6 Pull-to-Refresh

Поддержка обновления данных жестом `.refreshable { await vm.load() }`.

## 7. Функциональный модуль «Проекты»

### 7.1 Описание

Модуль управления проектами. Обеспечивает полный CRUD (создание, чтение, обновление, удаление).

### 7.2 Список проектов (ProjectsView)

**Навигация:**
- Вторая вкладка в TabView
- Иконка: `folder.fill`
- Заголовок: «Проекты»

**Визуальная структура:**
- Встроенное поле поиска (`.searchable`)
- Фильтр по статусу (меню в toolbar)
- Список карточек проектов
- Кнопка `+` создания проекта (только admin/manager)

### 7.3 Поиск

- Встроенный в NavigationStack (`.searchable`)
- Фильтрация по названию проекта (`localizedCaseInsensitiveContains`)
- Работает в реальном времени при вводе

### 7.4 Фильтрация по статусу

Меню с иконкой-фильтром в правой части toolbar:
- **Все** — показывает все проекты (фильтр снят)
- **Планирование** — `status == "planning"`
- **Активные** — `status == "active"`
- **На паузе** — `status == "on_hold"`
- **Завершённые** — `status == "completed"`

Иконка фильтра меняет вид при активном фильтре (заполненная / незаполненная).

### 7.5 Карточка проекта (ProjectListRow)

Компоненты карточки сверху вниз:
1. **Заголовок:** название (`.subheadline.semibold`) + бейдж приоритета
2. **Описание:** до 2 строк (`.caption`, серый)
3. **Прогрессбар:** `ProgressView(value:)`, зелёный при 100%, синий иначе
4. **Статистика:** `X/Y задач` (левый край) + бейдж статуса (правый край)
5. **Подвал:** иконка + имя менеджера | дата окончания (красная при просрочке)

**Стилизация:**
- Белый фон, закруглённые углы 14pt
- Тень: `shadow(color: .black.opacity(0.05), radius: 4, x: 0, y: 2)`
- Внутренние отступы: 16pt

### 7.6 Действия с проектом (admin/manager)

**Контекстное меню (долгое нажатие):**
- «Редактировать» (иконка: pencil)
- «Удалить» (иконка: trash, деструктивная)

**Свайп влево (swipeActions):**
- Красная кнопка «Удалить» (полный свайп разрешён)
- Оранжевая кнопка «Изменить»

### 7.7 Форма создания / редактирования (ProjectFormView)

**Секция «Основное»:**
| Поле | Тип ввода | Обязательное | Ограничения |

| Название проекта | `TextField` | Да | Не пустое (с trim) |
| Описание | `TextField` (многострочный, 3-6 строк) | Нет | — |

**Секция «Параметры»:**
| Поле | Тип ввода | Значение по умолчанию |
|---|---|---|
| Статус | `Picker` (Планирование/Активный/На паузе/Завершён) | Планирование |
| Приоритет | `Picker` (Низкий/Средний/Высокий/Критический) | Средний |

**Секция «Сроки»:**

| Дата начала | `DatePicker` | Нажатие на дату — сброс |

**Toolbar:**

### 7.8 Статусы проектов

| `medium` | Средний | Синий |
| `high` | Высокий | Оранжевый |

## 8. Функциональный модуль «Детальный вид проекта»

### 8.1 Описание

Полнофункциональный экран проекта с четырьмя режимами просмотра.

### 8.2 Шапка проекта (headerSection)

Всегда отображается над вкладками:
- **Строка 1:** Бейдж статуса | Бейдж приоритета | Дата окончания (с иконкой календаря)
- **Строка 2:** Сетка мини-статистик 4 колонки:

| Статистика | Цвет | Формула |
|---|---|---|
| Задач | Синий | Количество задач верхнего уровня (`parent_task_id == nil`) |
| Готово | Зелёный | Задачи со статусом `done` |
| Просроч. | Красный | Задачи с `isOverdue == true` |
| Прогресс | Синий | `(done_tasks / total_tasks) × 100%` |

- **Строка 3:** Общий прогрессбар проекта

### 8.3 Вкладки (Segmented Picker)

| Индекс | Название | Описание |
|---|---|---|
| 0 | Kanban | Горизонтальная доска по статусам |
| 1 | Список | Иерархическое дерево задач (WBS) |
| 2 | Гант | Диаграмма Ганта с временной шкалой |
| 3 | Участники | Список членов команды проекта |

### 8.4 Вкладка «Kanban»

**Структура:** горизонтальная прокрутка → 4 колонки (ширина 210pt каждая):

| Колонка | Статус | Описание |
|---|---|---|
| К выполнению | `todo` | Задачи, ожидающие начала |
| В работе | `in_progress` | Задачи в активной разработке |

## 10. Функциональный модуль «Управление задачами»

### 10.1 Описание

Система поддерживает полноценное управление задачами с **Work Breakdown Structure (WBS)** — декомпозицией работ на подзадачи любого уровня вложенности.

### 10.2 Иерархическая структура (WBS)

```
Проект
├── Задача 1 (верхний уровень)
│   ├── Подзадача 1.1
│   │   ├── Подзадача 1.1.1
│   │   └── Подзадача 1.1.2
│   └── Подзадача 1.2
├── Задача 2
│   └── Подзадача 2.1
└── Задача 3 (без подзадач)
```

**Ключевые свойства:**
- Глубина вложенности **не ограничена**
- Каждая подзадача является **полноценной задачей** со всеми атрибутами
- Подзадача может иметь свои подзадачи
- Связь реализована через поле `parent_task_id`

### 10.3 Агрегация прогресса

Прогресс родительской задачи вычисляется рекурсивно:

```
allDescendants = все потомки на всех уровнях (рекурсивный обход)
doneCount      = потомки со статусом "done"
totalCount     = общее количество потомков
progress       = doneCount / totalCount
```

Пример:
```
Задача 1 (progress: 2/5 = 40%)
├── Подзадача 1.1 (done) ✓
├── Подзадача 1.2 (in_progress)
│   ├── Подзадача 1.2.1 (done) ✓
│   └── Подзадача 1.2.2 (todo)
└── Подзадача 1.3 (todo)
```

### 10.4 Рекурсивная структура данных (TaskNode)

```swift
indirect enum TaskNode: Identifiable {
    case node(task: ProjectTask, children: [TaskNode])
}
```

Использование `indirect enum` позволяет описать рекурсивную структуру без ограничений глубины.

### 10.5 Создание подзадачи

**Точки входа:**
1. Меню `⋯` на любой задаче → «Добавить подзадачу»
2. Доступно на задачах **любого уровня** (в т.ч. на подзадачах)

**Процесс:**
1. Пользователь нажимает «Добавить подзадачу» в меню `⋯`
2. Открывается `TaskFormView` с заголовком «Новая подзадача»
3. В шапке формы отображается: «Подзадача для: [название родителя]»
4. Этап (`stage_id`) наследуется от родительской задачи
5. Пользователь заполняет все поля полноценной задачи
6. При сохранении: `parent_task_id = parentTask.id`

### 10.6 Форма задачи / подзадачи (TaskFormView)

Единая форма для создания задачи верхнего уровня, создания подзадачи и редактирования.

**Заголовок формы:**
| Контекст | Заголовок |
|---|---|
| Новая задача | «Новая задача» |
| Новая подзадача | «Новая подзадача» |
| Редактирование | «Редактировать задачу» |

**Секция «Родительская задача» (только для подзадач):**
- Иконка `list.bullet.indent` (синяя)
- Подпись: «Подзадача для:»
- Название родительской задачи (полужирный)

**Секция «Основное»:**
| Поле | Тип | Обязательное |
|---|---|---|
| Название | `TextField` | Да |
| Описание | `TextField` (многострочный, 3-6 строк) | Нет |

**Секция «Параметры»:**
| Поле | Тип | Значение по умолчанию |
|---|---|---|
| Статус | `Picker` (К выполнению / В работе / На ревью / Готово) | К выполнению |
| Приоритет | `Picker` (Низкий / Средний / Высокий / Критический) | Средний |

## 12. Функциональный модуль «KPI / Метод освоенного объёма»

### 12.1 Описание

Модуль реализует мониторинг проектов по методологии **Earned Value Management (EVM)** — международному стандарту измерения производительности проектов (ANSI/EIA-748).

### 12.2 Навигация

- Четвёртая вкладка в TabView
- Иконка: `chart.line.uptrend.xyaxis`
- Заголовок: «KPI — EVM»

### 12.3 Выбор проекта

`Picker` в стиле `.menu`:
- Показывает проекты со статусами `active` и `planning`
- При смене проекта загружаются соответствующие KPI-снимки
- По умолчанию выбран первый проект

### 12.4 KPI-индикаторы

Сетка 2×2 карточек `KPICard`:

| Показатель | Полное название | Формула | Норма | Цвет при норме | Цвет при отклонении |
|---|---|---|---|---|---|
| **SPI** | Schedule Performance Index | EV / PV | ≥ 1.0 | Зелёный + ✓ | Красный + ⚠ |
| **CPI** | Cost Performance Index | EV / AC | ≥ 1.0 | Зелёный + ✓ | Красный + ⚠ |
| **CV** | Cost Variance | EV − AC | ≥ 0 | Зелёный + ✓ | Красный + ⚠ |
| **SV** | Schedule Variance | EV − PV | ≥ 0 | Зелёный + ✓ | Красный + ⚠ |

**Визуальное оформление KPICard:**
- Рамка: зелёная (норма) или красная (отклонение), 1.5px
- Иконка: `checkmark.circle.fill` или `exclamationmark.circle.fill`
- Подпись: «В норме» или «Отклонение»
- Тень + закруглённые углы 12pt

### 12.5 EVM-график

**Тип:** линейный график (`LineMark`)  
**Библиотека:** Swift Charts

**Три линии:**
| Линия | Метрика | Цвет | Стиль |
|---|---|---|---|
| PV | Planned Value | Серый | Пунктирная (dash: [4]) |
| EV | Earned Value | Синий | Сплошная |
| AC | Actual Cost | Красный | Сплошная |

**Маркеры:** кружки на каждой точке данных  
**Ось X:** дата снимка  
**Ось Y:** значение в рублях (BYN)  
**Высота:** 220pt  
**Легенда:** автоматическая

### 12.6 Прогнозные показатели

| Показатель | Полное название | Формула | Описание |
|---|---|---|---|
| **EAC** | Estimate at Completion | BAC / CPI | Прогнозная итоговая стоимость проекта |
| **ETC** | Estimate to Complete | EAC − AC | Сколько ещё потребуется вложить |
| **BAC** | Budget at Completion | Задан | Исходный утверждённый бюджет |

Отображаются двумя `StatCard`:
- EAC — оранжевый, подпись «Прогноз по завершению (BYN)»
- ETC — фиолетовый, подпись «Оставшиеся затраты (BYN)»

### 12.7 Интерпретация показателей

| SPI/CPI | Интерпретация |
|---|---|
| > 1.0 | Проект опережает план (экономия) |
| = 1.0 | Проект идёт точно по плану |
| < 1.0 | Проект отстаёт (перерасход) |

| CV/SV | Интерпретация |
|---|---|
| > 0 | Положительное отклонение (экономия/опережение) |
| = 0 | Точно по плану |
| < 0 | Отрицательное отклонение (перерасход/отставание) |

## 18. Ограничения и перспективы развития

### 18.1 Текущие ограничения

| Ограничение | Описание |
|---|---|
| Хранение данных | In-memory — данные не сохраняются между перезапусками |
| Многопользовательский доступ | Не реализован — данные локальны для устройства |
| Комментарии к задачам | Модель определена, UI не реализован |
| Вложения и файлы | Не реализовано |
| Push-уведомления | Не реализовано |
| Управление этапами | Этапы предзаполнены, UI для создания/редактирования отсутствует |
| Управление участниками | Участники предзаполнены, UI для добавления/удаления отсутствует |

### 18.2 Перспективы развития

| Направление | Описание |
|---|---|
| Серверный бэкенд | Подключение REST API (Node.js + PostgreSQL) |
| Синхронизация в реальном времени | WebSocket для совместной работы |
| Персистентность | SwiftData / Core Data для локального хранения |
| Уведомления | Push-уведомления о дедлайнах и назначениях |
| Файловые вложения | Прикрепление документов к задачам |
| Комментарии | Обсуждение задач в комментариях |
| Отчёты | Экспорт данных в PDF/Excel |
| Управление этапами | CRUD для этапов проекта |
| Управление командой | Добавление/удаление участников |
| Временные затраты | Time-tracking с таймером |
